
/*很开心大家能够打开我的项目文档，让我们一起回顾一下RCS战队 RoboMaster26赛季我们队伍步兵机器人在嵌入式方面有哪些稳定成熟的技术栈，又有哪些部分需要改进*/

我们来简单回顾一下RoboMaster嵌入式部分需要完成的部分技术栈
1.FreeRTOS调度系统
2.can通信
3.串口通信
4.电机控制
5.运动学控制

1.FreeRTOS调度系统
已完成部分
1.稳定的电机控制周期与电机力矩(电流)计算周期
2.稳定的基于FreeRTOS自带的消息队列接收

待完善部分
1.在任务被阻塞时没有将单个任务reset的功能
2.没有实现task之间的通信，这会导致部分数据出现同时读写的情况，导致数据出现问题
3.部分任务的优先级可能还需要调整
/*建议队内增加一个开门狗复位的机制*/

2.can通信
已完成部分
1.大疆电机的基本控制
2.与上位机NX的基本通信
3.达妙电机的基本控制

待完善部分
CAN通信是一个相对复杂的通信手段，需要大家去深入了解
1.因为本人在达妙电机的解码和发送数据时加了互斥锁，导致了单片机在线而电机侧断电时会出现达妙解码数据从此就再也不更新的情况，从而在场上出现过无法使能的情况
2.大疆电机的互斥锁问题
3.can线与下位机断联导致机器人失控的情况，尤其是你们日后要做的轮足机器人，本人想到的方法是在上位机can通信部分加一个冗余保护，也就是如果检测到下位机断联就失能全部电机

3.串口通信
已完成部分
1.稳定的收发
2.收数据部分走的是DMA

待完善部分
1.发送目前走的还是普通的阻塞形的发送，希望改成DMA
2.串口通信CRC部分大家要重视，之前队里出现过陀螺仪数据收不到把CRC注释就能收到的情况，这种问题以后不要出现，一定要加CRC，要去排查问题而不是逃避

4.电机控制
已完成部分
1.以IMU数据为准的云台电机控制
2.小陀螺状态下的电机的稳定控制

待完善部分
1.我不建议大家云台部分去走所谓的滑膜控制等等更复杂的算法，因为一个稳定的嵌入式底层+PID+数据预处理（前馈）等完全就足够处理我们正常的需求

5.运动学控制
已完成部分
1.底盘解算
2.云台保持
3.功率控制

待完善部分
1.在实战中我发现了功率控制不能完全吃满升级后带来的性能提升，也有可能是电机在功率到达一定数值后每提升一点转速都需要更更巨大的功率的可能性，这个需要排查
2.功率控制虽然步兵赛场上从来没出过问题，但是我还是觉得有点奇怪，因为不上超电的时候还是很容易超功率，按照我的算法按道理不应该出现这种情况，建议重写
